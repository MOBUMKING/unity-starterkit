이 프로젝트가 무엇인가요?

Arrows 스타일의 레벨 선택형 모바일 게임을 빠르게 만들 수 있도록 설계된 범용 스타터킷입니다.

왜 만들었나? — 모바일 게임 프로토타입을 만들 때마다 매번 비슷한 골격(씬 구성, 진행 저장, 화면 전환, 안전 영역 처리)을 새로 작성하는 비용을 줄이기 위해, 1~20 스테이지 진행형 게임의 공통 기반을 미리 갖춘 시작점을 만들었습니다. 실제 게임 콘텐츠(퍼즐·리듬·아케이드 등)만 InGamePanel에 추가하면 그대로 출시 가능한 구조를 목표로 합니다.

핵심 게임 루프

앱 시작 → Boot 씬 (저장 데이터 불러오기) → 메인 화면 (타이틀 + 스테이지 번호 + 플레이 버튼) → 인게임 화면 (스테이지 번호 + 게임 영역 + 클리어/실패 버튼) ├─ 클리어 → 클리어 연출 → 다음 스테이지 메인 화면 │ └─ 20스테이지 클리어 → 게임 완료 화면 └─ 실패 → 실패 연출 → 동일 스테이지 메인 화면 복귀

핵심 설계: "스테이지 진행 → 판정 → 연출 → 다음 스테이지"라는 4단 사이클은 거의 모든 레벨 기반 모바일 게임의 공통 구조입니다. 이 사이클을 컨트롤러(GameFlowController)와 데이터(SaveManager·StageManager)로 분리하면, 실제 게임 규칙이 바뀌어도 같은 골격을 그대로 쓸 수 있습니다.

프로토타입 단계에서는 인게임 화면에 "클리어 / 실패 버튼"으로 판정을 수동 트리거합니다. 실제 게임플레이 구현 시 이 버튼은 제거되고, 동일한 판정 메서드를 게임플레이 시스템이 호출합니다.

M1 프로젝트 기반 & 씬 구조

📁 폴더 구조 — 왜 _Project로 격리?

Assets/_Project/
├── Scripts/  (Managers, UI, Data)
├── Scenes/   (Boot, Main)
├── ScriptableObjects/
├── UI/  Prefabs/  Animations/
└── Materials/  Editor/

Asset Store 패키지·Unity 예약 폴더(Plugins, Resources 등)와 경로 충돌 방지. _ 접두사로 프로젝트 코드를 폴더 트리 최상단에 고정.

🎨 Canvas 설정 — 1080×1920 / Match 0.5

  • 1080×1920: 모바일 세로 표준 해상도
  • Match 0.5: 가로/세로 비율 변화에 균일 대응 — 어떤 비율(16:9 / 19.5:9 / 20:9)이든 가운데 기준으로 자연스럽게 늘어남
  • SafeArea GameObject를 Canvas 자식으로 미리 배치 → M4에서 SafeAreaHandler 부착
  • 5개 패널(Main/InGame/Clear/Fail/GameClear) 골격 비활성화 상태로 사전 배치

왜 이걸 가장 먼저? Canvas Scaler·Safe Area를 나중에 적용하면 이미 배치된 UI 좌표가 모두 어긋나 재작업해야 합니다. 해상도 기준을 가장 먼저 고정하는 것이 UI 작업 비용을 가장 크게 줄입니다.

📦 StageDataSO — 필드별 상세

Assets/_Project/ScriptableObjects/Stage01.asset ~ Stage20.asset 총 20개 사전 생성.

필드역할사용처 / 확장
StageNumber: int 스테이지 고유 식별자(1~20). 인스펙터에서 직접 입력 UI 라벨, 저장 데이터 비교, AdvanceToNextStage 분기 / 100+ 스테이지 확장 시 StageManager 클램프 범위만 조정

ClearAnimationType: enum 스테이지 클리어 시 재생할 연출 종류 M6 AnimationController가 enum으로 분기 / Boss·MiniGame 등 새 값 추가 시 enum 한 줄만 추가

확장 팁: AudioClip(BGM), Sprite(배경 이미지), 보상 골드 등 스테이지별 추가 데이터를 필드로 추가하면 코드 수정 없이 스테이지마다 다른 콘텐츠 구성이 가능합니다.

🗂 StageManager — 메서드별 상세

싱글턴 / DontDestroyOnLoad 적용 / Assets/_Project/Scripts/Managers/StageManager.cs

API역할호출처 / 확장
CurrentStageNumber get-only 현재 도전 스테이지 번호. 외부 직접 변경 차단 — 오직 SetCurrentStage()로만 갱신 UIManager가 화면 표시 시 / GameFlowController가 클리어 판정 시 참조
SetCurrentStage(int) 특정 번호로 점프 설정. Mathf.Clamp(1, 20)로 자동 보정 — 잘못된 값을 넣어도 안전 BootLoader가 Load() 결과 +1로 호출 / 디버그용 스테이지 점프 메뉴에서 그대로 사용
AdvanceToNextStage() 다음 스테이지로 +1. 단, IsLastStage()가 true면 무시 — 20에서 21로 가는 사고 방지 내장 GameFlowController.OnStageClear() 내부 / 호출처는 마지막 단계 분기를 따로 짤 필요 없음
IsLastStage(): bool 현재 번호 ≥ 20 인지 반환 게임 완료 화면(GameClearPanel) 분기 판단 / 광고 노출, 엔딩 이벤트 트리거 등 다양한 분기에 활용

💾 SaveManager — 메서드별 상세

PlayerPrefs 키 "MaxClearedStage"를 외부에 노출하지 않는 캡슐화 매니저

API역할호출처 / 확장
Save(int maxClearedStage) 클리어한 최고 스테이지 번호를 PlayerPrefs에 저장. SetInt + Save를 즉시 호출하여 휘발성 제거 GameFlowController.OnStageClear() 직후 / M7-D에서 SaveAsync(int) UniTask 옵션 추가 예정
Load(): int 저장값 반환. 미저장 시 기본값 0 — 첫 실행도 분기 없이 자연 처리 BootLoader가 앱 진입 시 호출 / 0이면 1스테이지부터, N이면 (N+1)스테이지부터 자동 시작
Reset() 키 삭제 + Save. 진행 데이터를 완전 초기화 UIManager.OnRestartButtonClick() / GameClearPanel "처음부터 다시" 버튼

왜 캡슐화했나? PlayerPrefs 키 이름이 외부 코드에 흩뿌려져 있으면, 저장 방식을 JSON 파일·클라우드 동기화로 바꿀 때 모든 호출처를 수정해야 합니다. SaveManager 내부에서만 키를 다루므로 저장 백엔드 교체 시 외부 영향 0입니다.

M3 Boot 씬 진입 & 초기화 흐름

🚀 BootLoader (Boot 씬 전용 진입점)

앱이 시작될 때 가장 먼저 실행되는 컴포넌트. 저장 데이터를 복원하고 Main 씬으로 전환한 뒤 자기 자신은 폐기됩니다 — "초기화만 담당하고 사라지는" 일회성 진입점.

Awake() — CancellationTokenSource 초기화 ↓ InitializeAsync(CancellationToken) — UniTask 비동기 ↓ SaveManager.Load() — MaxClearedStage 읽기 ↓ StageManager.SetCurrentStage(maxCleared + 1) ↓ SceneManager.LoadSceneAsync("Main").ToUniTask() ↓ OnDestroy() — _cts.Cancel() / Dispose()

설계 의도: 진입점을 명확히 분리해야 UIManager가 정확한 스테이지 번호로 화면을 그릴 수 있습니다. 다음 슬라이드에서 BootLoader의 설계 결정들을 자세히 설명합니다.

🚀 BootLoader — 설계 결정 풀어보기

설계 포인트왜 이렇게 했나확장 시 활용
비싱글턴 / DontDestroyOnLoad 미적용 Boot 씬 전용 — 초기화가 끝나면 씬과 함께 자동 폐기. 메모리·중복 인스턴스 부담 없음 새 진입점이 필요하면 BootLoader 자체에 단계 추가만 하면 됨 (별도 매니저 불필요)
UniTask + CancellationToken 코루틴 대비 GC 할당이 거의 없고, 도중 씬 종료/앱 종료 시 비동기 작업을 안전하게 끊을 수 있음 패치 다운로드 / 광고 SDK 초기화 / 로컬라이제이션 로드 같은 비동기 단계를 그대로 추가 가능
Load() → SetCurrentStage(+1) 저장값은 "클리어한 최고 번호"라서 다음 도전 번호는 +1. 이 변환을 BootLoader가 단 한 군데에서 수행 저장 포맷이 바뀌어도 변환 로직은 BootLoader에서만 수정 — UIManager는 영향 없음
SceneManager.LoadSceneAsync("Main") Main 씬으로 비동기 전환. 동기 로드 대비 진행률 노출 / 로딩 화면 추가가 쉬움 로딩 진행바, 스플래시 페이드 등을 BootLoader 안에서 그대로 확장

대표적 확장 시나리오: Firebase Remote Config 동기화, IAP 카탈로그 로드, AB 테스트 분기 결정 등을 모두 InitializeAsync 안에서 처리 → 게임 화면이 켜지기 전에 모든 외부 의존이 준비됩니다.

M4 UIManager & 화면 전환 골격

UI를 5개 패널(Main / InGame / Clear / Fail / GameClear)로 분리하고, 단일 매니저(UIManager)가 활성/비활성을 제어하는 단순 구조. 씬 전환 없이 패널만 바꾸므로 전환 비용이 거의 0이고, 패널 간 데이터 공유가 간단합니다.

🖼 UIManager (Main 씬 싱글턴)

  • Inspector에 5개 패널 GameObject 할당
  • Awake에서 PanelType→GameObject Dictionary 구성
  • ShowPanel(PanelType)로 단일 패널 활성화
  • 스테이지 텍스트 갱신, 버튼 이벤트 메서드 보유

📐 SafeAreaHandler

  • Canvas 자식 SafeArea GameObject에 부착
  • Awake에서 Screen.safeArea를 0~1 앵커로 변환
  • 노치/펀치홀/홈바 영역을 자동으로 회피
  • 모든 패널은 SafeArea의 자식으로 두어 자동 적용

다음 2장에서 UIManager와 SafeAreaHandler를 메서드 단위로 풀어 설명합니다.

🖼 UIManager — 메서드 / 필드 상세

API역할호출처 / 확장
_panelMap Dict PanelType enum → GameObject 매핑. Awake에서 5쌍 등록 새 패널 추가 시: enum 값 추가 + Inspector 필드 + Dictionary 한 줄. 호출 코드는 변경 불필요
ShowPanel(PanelType) 지정 패널만 활성화 / 나머지 비활성화 BootLoader에서 Main 씬 진입 시 / GameFlowController에서 클리어·실패·완료 분기 시 / M7-C에서 O(1) 캐시로 최적화 예정
RefreshStageDisplay() StageManager에서 현재 번호를 읽어 메인·인게임 패널의 TMP_Text 동시 갱신 BootLoader 진입 직후, 스테이지 변경(+1) 직후 호출 / 다국어 시 STAGE 접두사를 LocalizationKey로 변경
OnPlayButtonClick() 플레이 버튼이 호출. RefreshStageDisplay → ShowPanel(InGame) Inspector OnClick UnityEvent로 직접 연결 / 효과음·BGM 트리거 추가 가능

📐 SafeAreaHandler — 동작 원리

🔄 변환 과정

  • Screen.safeArea는 픽셀 단위 Rect (예: 좌하 0, 100 / 우상 1080, 1820)
  • 이를 화면 전체(Screen.width × height)로 나눠 0~1 비율 좌표로 변환
  • RectTransform의 anchorMin / anchorMax에 적용 → 자식들은 자동으로 안전 영역 안에 그려짐

📍 부착 위치 / 사용 패턴

  • Canvas 직속 자식 GameObject "SafeArea" 하나에만 부착
  • 모든 화면 패널(Main/InGame/Clear/Fail/GameClear)은 SafeArea의 자식으로 둠
  • 각 패널은 stretch 앵커(0,0)~(1,1)로 두면 SafeArea 영역 안에서 꽉 채워짐
  • 코드 추가 작성 불필요 — 컴포넌트 부착만으로 모든 디바이스 대응

현재 한계 → M7-C에서 보강: Awake에서 한 번만 적용하므로 화면 회전·iPad 멀티 윈도우는 미대응. 회전 게임은 OnRectTransformDimensionsChange 또는 매 프레임 Screen.safeArea 비교를 추가해 동적 모니터링하도록 확장합니다.

지금 동작하는 전체 흐름 (M5 완료)

[앱 실행] ↓ Boot.unity — BootLoader.Awake() ↓ SaveManager.Load() → "MaxClearedStage" 읽기 ↓ StageManager.SetCurrentStage(N+1) ↓ SceneManager.LoadSceneAsync("Main") ↓ Main.unity — UIManager.Awake() / Start() ↓ SafeAreaHandler가 디바이스 노치/홈바 자동 보정 ↓ RefreshStageDisplay() → "STAGE N" 텍스트 갱신 ↓ ShowPanel(PanelType.Main) → 메인 화면 표시 ↓ [플레이 버튼] → UIManager.OnPlayButtonClick() → ShowPanel(InGame) [클리어 버튼] → GameFlowController.OnStageClear() ├── SaveManager.Save(N) → 저장 ├── IsLastStage==true → ShowPanel(GameClear) └── else → AdvanceToNextStage → RefreshStageDisplay → ShowPanel(Main) [실패 버튼] → GameFlowController.OnStageFail() └── RefreshStageDisplay → ShowPanel(Main) (스테이지 번호 유지) [처음부터 다시] → UIManager.OnRestartButtonClick() └── SaveManager.Reset → SetCurrentStage(1) → ShowPanel(Main)

✅ M5 완료 — 클리어/실패 버튼이 GameFlowController와 연결되어 첫 스테이지 → 20스테이지 → 게임 완료까지 끊김 없이 동작하는 완전 MVP가 완성되었습니다. 다음 M6에서 DOTween 연출을 삽입합니다.

기능 단위로 어떻게 활용하나

스타터킷 사용 가이드

시스템 의존 관계도

BootLoader (Boot 씬 전용)
  ├── SaveManager     ✅
  └── StageManager    ✅

GameFlowController     ✅
  ├── StageManager    ✅
  ├── UIManager       ✅
  └── SaveManager     ✅

UIManager (Main 씬)    ✅
  ├── StageManager    ✅
  ├── SaveManager     ✅
  └── AnimationController (M6 예정)

SafeAreaHandler        ✅ (의존 없음)
StageManager           ✅ (의존 없음, 독립)
SaveManager            ✅ (의존 없음, 독립)

단방향 의존 규칙의 의미: 상위(GameFlowController) → 하위(StageManager) 방향만 허용. StageManager가 GameFlowController를 호출하면 안 됨. 이 규칙을 지키면 ① 의존 사이클이 생기지 않아 컴파일 단위 분리가 가능하고, ② 하위 매니저는 단독 테스트가 가능하며, ③ 상위 매니저를 교체해도 하위는 영향 없습니다.

🗂 StageManager 활용법

언제든지 StageManager.Instance로 접근 — DontDestroyOnLoad라 씬 전환 후에도 유지

// 현재 도전 스테이지 번호 (1~20, 외부 직접 변경 불가)
int n = StageManager.Instance.CurrentStageNumber;

// 현재 스테이지 데이터(SO) — 인덱스 안전 검사 내장
StageDataSO data = StageManager.Instance.GetCurrentStageData();
string name = data.StageName;
AnimationType clearAnim = data.ClearAnimationType;

// 다음 스테이지로 진행 (마지막이면 자동 무시 — 호출처에서 분기 불필요)
StageManager.Instance.AdvanceToNextStage();

// 마지막 스테이지 여부로 게임 완료 분기 판단
if (StageManager.Instance.IsLastStage())
    UIManager.Instance.ShowPanel(PanelType.GameClear);

// 특정 스테이지로 점프 (1~20 자동 클램프, 잘못된 값 안전)
StageManager.Instance.SetCurrentStage(5);

💾 SaveManager 활용법

PlayerPrefs를 캡슐화 — 키 이름을 외부에 노출하지 않음

// 클리어한 최고 스테이지 저장 (게임 클리어 직후, 연출 재생 전에 호출)
SaveManager.Instance.Save(maxClearedStage: 5);

// 저장된 진행 상태 불러오기 (BootLoader가 앱 진입 시 자동 호출)
int saved = SaveManager.Instance.Load();   // 미저장 시 0 반환

// 진행 데이터 초기화 ("처음부터 다시" 버튼)
SaveManager.Instance.Reset();

확장 패턴: 별점, 게임 시간, 보유 골드 등 새 키를 추가할 때는 SaveManager 안에서만 PlayerPrefs를 다루도록 메서드를 추가하세요. 외부 코드는 키 이름을 몰라도 되며, 저장 백엔드를 JSON·클라우드로 바꿀 때도 SaveManager 내부만 수정하면 됩니다.

🖼 UIManager 활용법

// 화면 전환 (Main 씬 어디서든 한 줄로)
UIManager.Instance.ShowPanel(PanelType.InGame);
UIManager.Instance.ShowPanel(PanelType.Clear);
UIManager.Instance.ShowPanel(PanelType.GameClear);

// 스테이지 텍스트 강제 갱신 (스테이지 +1 직후)
UIManager.Instance.RefreshStageDisplay();

// 버튼 OnClick 인스펙터에 직접 연결되는 메서드
//   - 플레이 버튼   → OnPlayButtonClick()    (Main → InGame 전환)
//   - 처음부터 다시 → OnRestartButtonClick() (Reset → Main 복귀)

새 패널 추가 절차 (3단계):

  1. PanelType enum에 새 값 추가 (예: Settings)
  2. UIManager Inspector에 새 GameObject 슬롯 추가 후 패널 할당
  3. Awake의 _panelMap에 한 줄 매핑 추가

호출 코드는 ShowPanel(PanelType.Settings)로 그대로 호출 가능 — 분기 없음.

📐 SafeAreaHandler 활용법

코드 호출이 필요 없는 "부착만 하면 끝" 형태의 컴포넌트입니다. Canvas 자식 SafeArea GameObject에 부착하고, 모든 패널을 그 자식으로 두기만 하면 디바이스의 노치·펀치홀·홈바를 자동으로 회피합니다.

적용 대상상황결과
iPhone X 이상상단 노치 / 하단 홈바UI가 노치·홈바 영역을 침범하지 않음
안드로이드 펀치홀상단 카메라 영역해당 영역 자동 회피
일반 디바이스 (베젤)safeArea = 전체 화면변환 결과 (0,0)~(1,1) — 영향 없음
iPad 멀티 윈도우화면 비율 동적 변경현재 미대응 → M7-C에서 동적 모니터링 추가 예정

📦 StageDataSO 활용법

Unity ScriptableObject 기반이라 코드 수정 없이 인스펙터만으로 데이터 추가/변경이 가능합니다. 디자이너·기획자가 직접 스테이지 구성을 바꿀 수 있는 구조.

작업방법결과
새 스테이지 추가 메뉴: Create > Stage Data SO → Inspector에서 번호·이름 입력 StageManager._stageData 배열에 드래그 → 즉시 사용 가능
기존 스테이지 데이터 수정 Stage##.asset 선택 → Inspector에서 필드 변경 플레이 모드 재시작만으로 즉시 반영, 코드 빌드 불필요
새 데이터 필드 추가 (예: AudioClip) StageDataSO.cs에 [SerializeField] private AudioClip _bgm 추가 + 프로퍼티 20개 에셋 인스펙터에 자동으로 새 필드 노출 — 기획자가 채워넣음

🎮 스타터킷 활용 시나리오

공통: UI 골격·진행 저장·화면 전환·연출 삽입 위치는 그대로, InGamePanel 내부에 게임 콘텐츠만 추가

만들고 싶은 게임바꿀 부분활용 포인트
퍼즐 게임 (라인 잇기 등) InGamePanel에 퍼즐 보드 추가, 클리어 조건 충족 시 OnStageClear() 호출 StageDataSO에 퍼즐 데이터(맵 크기·시작 배치) 필드 추가 — 스테이지마다 다른 퍼즐을 인스펙터에서 구성
리듬 게임 (스테이지별 곡) StageDataSO에 AudioClip / 노트 데이터 추가, InGamePanel에 노트 시스템 연결 곡 끝까지 일정 점수 이상 → OnStageClear / 미스 X회 → OnStageFail
아케이드 (피하기·점프) InGamePanel에 캐릭터·장애물 시스템, 사망 시 OnStageFail 호출 20개 스테이지 = 20단계 난이도. StageDataSO에 장애물 속도/패턴 필드 추가
퀴즈 게임 StageDataSO에 질문/보기 필드 추가, 정답 시 OnStageClear 20문제 한 세트로 출제 / 오답 시 OnStageFail로 동일 문제 재시도

🎮 GameFlowController 활용법

// 클리어 판정 발생 시 (버튼 또는 실제 게임플레이 시스템에서 호출)
GameFlowController.Instance.OnStageClear();
// 내부 동작:
//   1. SaveManager.Instance.Save(currentStageNumber)  — 즉시 저장
//   2. IsLastStage() → true  : UIManager.ShowPanel(PanelType.GameClear)
//             → false : AdvanceToNextStage → RefreshStageDisplay → ShowPanel(Main)

// 실패 판정 발생 시
GameFlowController.Instance.OnStageFail();
// 내부 동작:
//   스테이지 번호 유지, 저장 없음
//   RefreshStageDisplay → ShowPanel(PanelType.Main)  (동일 스테이지로 복귀)

프로토타입 버튼 제거 방법: 실제 게임플레이 규칙을 구현한 후, InGamePanel의 클리어/실패 버튼을 제거하고 게임플레이 시스템이 동일한 OnStageClear() / OnStageFail()을 호출하도록 하면 됩니다. GameFlowController 내부 로직은 전혀 변경 불필요 — 이 설계가 스타터킷의 핵심 재사용 포인트입니다.
