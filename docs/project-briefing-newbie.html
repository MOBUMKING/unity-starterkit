    개요 1/3

    이 프로젝트가 무엇인가요?

    레벨 선택형 모바일 게임을 만들 때마다 반복되는 공통 작업들 — 화면 구성, 진행 저장, 화면 전환, 노치·홈바 대응 — 을 미리 만들어 둔 시작점입니다.

    왜 만들었나? — 새 게임을 만들 때마다 "1스테이지부터 20스테이지까지 진행하고, 끝나면 저장되고, 다시 켜면 이어서 시작되는" 같은 뼈대를 처음부터 다시 짜는 것은 시간 낭비입니다. 이 프로젝트는 그 뼈대를 미리 완성해 둔 것입니다. 실제 게임 내용(퍼즐·리듬·아케이드 등)만 중간에 끼워 넣으면 됩니다.

    개요 2/3

    게임이 어떻게 돌아가나요?

    앱 시작 ↓ 초기화 담당자(BootLoader)가 실행됨 ↓ 저장된 진행 상황 불러오기 → "이전에 5스테이지까지 했구나" ↓ 다음 도전 스테이지 = 6으로 설정 ↓ 메인 화면으로 이동 ↓ 메인 화면 — "STAGE 6" 표시 + 플레이 버튼 ↓ 플레이 버튼 클릭 ↓ 게임 화면 — 실제 게임 진행 (현재는 테스트용 버튼으로 대체) ├─ 클리어! → 진행 상황 저장 → 다음 스테이지(7) 메인 화면으로 │ └─ 20스테이지 클리어 → 🎉 게임 완료 화면 └─ 실패… → 저장 없음 → 같은 스테이지(6) 메인 화면으로 복귀

    핵심 설계 포인트: "진행 → 판정 → 저장 → 다음 화면"이라는 4단 사이클은 거의 모든 레벨 기반 모바일 게임의 공통 구조입니다. 이 사이클을 미리 완성해 두면, 실제 게임 규칙(퍼즐 풀기, 노래 맞추기 등)이 바뀌어도 뼈대는 그대로 사용할 수 있습니다.

    ⚠️ 지금 단계에서는 "클리어 버튼 / 실패 버튼"을 직접 눌러서 판정을 테스트합니다. 실제 게임이 추가되면 이 버튼은 제거됩니다.

    PART 1 · M1

    1단계 완료 프로젝트 기반 만들기

    📁 파일 정리 방식 — 왜 "_Project" 폴더?

    • 모든 게임 파일을 _Project 폴더 하나에 모아서 관리
    • 외부에서 구매하거나 다운로드하는 도구들과 섞이지 않도록 분리
    • Scripts(코드) / Scenes(화면) / Data(데이터) 등 역할별로 분류
    • _ 접두사를 붙이면 폴더 목록 맨 위에 고정되어 찾기 쉬움

    📱 화면 해상도 기준 설정 — 왜 1080×1920?

    • 1080×1920: 모바일 세로 화면 표준 크기
    • 가로·세로 비율이 달라도 화면 중심 기준으로 자연스럽게 대응
    • 16:9 / 19.5:9 / 20:9 어떤 폰이든 UI가 잘려 보이지 않음
    • 노치·홈바 대응을 위한 안전 영역도 이 단계에서 미리 자리 잡아 둠
    • 5개 화면 골격(메인·게임·클리어·실패·게임완료)을 미리 배치

    왜 가장 먼저 했나? — 화면 해상도 기준은 집을 지을 때 "기초공사"와 같습니다. 나중에 바꾸면 이미 배치된 모든 UI 위치가 틀어져서 전부 다시 해야 합니다. 이 설정을 제일 먼저 확정해야 이후 모든 UI 작업이 올바른 위치에 자리를 잡을 수 있습니다.

    PART 1 · M2 (1/2)

    2단계 완료 데이터 & 저장 시스템 개요

    왜 화면보다 데이터를 먼저? — 데이터 담당자들이 먼저 완성되어야 화면을 만들 때 "스테이지 번호 어디서 가져오지?" 같은 막힘이 없습니다. 설계도(데이터)를 먼저 그리고 벽(화면)을 세우는 것과 같습니다.

    📦 스테이지 데이터 파일

    각 스테이지의 정보(번호, 이름, 연출 종류)를 저장하는 엑셀 시트 같은 파일. 20개가 미리 만들어져 있어, 개발자가 코드를 수정하지 않아도 기획자가 직접 내용을 바꿀 수 있습니다.

    🗂 진행 상황 기록원

    게임 내 어디서든 "지금 몇 번째 스테이지야?"를 물어볼 수 있는 1개만 존재하는 관리자. 1~20 범위를 벗어나면 자동으로 보정합니다.

    💾 세이브 파일 담당자

    게임 진행 상황을 저장·불러오는 전담 담당자. 앱을 껐다 켜도 어디까지 했는지 기억합니다. 저장 방식을 바꿔도 다른 코드는 영향받지 않습니다.

    다음 슬라이드에서 각 담당자의 역할을 자세히 설명합니다.

    PART 1 · M2 (2/2)

    2단계 — 각 담당자 역할 상세

    담당자 이름하는 일실생활 비유
    스테이지 데이터 파일
    StageDataSO    		 스테이지 번호·이름·클리어 연출 종류·실패 연출 종류를 파일에 저장. 20개 파일이 미리 생성되어 있음. 인스펙터(설정 창)에서 클릭 한 번으로 값 변경 가능 영화 시나리오처럼, 각 "스테이지 1편·2편·…" 내용이 파일로 정리되어 있어 기획자가 직접 수정 가능
    진행 상황 기록원
    StageManager    		 현재 스테이지 번호를 기억. 다음 단계로 넘기기, 특정 단계로 이동, 마지막 단계인지 확인하는 기능 제공. 게임 전체에서 딱 1개만 존재 회사의 진도 관리 담당자처럼, "우리 팀은 지금 몇 번 업무 중이야?"를 언제든 물어볼 수 있음
    세이브 파일 담당자
    SaveManager    		 클리어한 최고 스테이지 번호를 폰의 메모장(PlayerPrefs)에 저장. 불러오기(앱 재시작 시), 초기화("처음부터 다시" 버튼)도 담당. 저장 방식 변경 시 이 파일만 수정하면 됨 가계부 담당자처럼, 저장·불러오기·초기화는 이 사람을 통해서만 처리. 다른 팀원은 저장 방식을 몰라도 됨

    왜 각 담당자를 분리했나? — 저장 방식을 "폰 내부 메모장"에서 "클라우드 서버"로 바꿔도 세이브 파일 담당자 내부만 수정하면 됩니다. 다른 담당자들은 "저장해줘", "불러와줘"만 말할 뿐 저장 방식을 알 필요가 없습니다.

    PART 1 · M3 (1/2)

    3단계 완료 앱 시작 초기화 흐름

    🚀 초기화 전문가 (BootLoader) — "사라지는 담당자"

    앱이 시작될 때 딱 한 번 실행되고, 역할이 끝나면 스스로 사라지는 진입점. 로딩 화면에서 뒤에서 조용히 일하는 담당자입니다.

    앱 켜짐 → "로딩 중…" 화면(Boot 화면) ↓ 초기화 전문가(BootLoader) 실행 ↓ 세이브 파일 담당자에게 물어봄 → "저장된 진행 상황 있어?" ↓ 있으면: "5단계까지 클리어했구나 → 다음 도전 = 6단계" ↓ 없으면: "처음 시작이구나 → 1단계부터" ↓ 진행 상황 기록원에게 현재 스테이지 전달 ↓ 메인 화면으로 이동 (이 순간 초기화 전문가는 사라짐) ↓ 메인 화면에서 "STAGE 6" 표시

    왜 "사라지는 담당자"인가? — 초기화는 딱 한 번만 필요합니다. 이후에는 메인 화면이 알아서 동작하기 때문에 초기화 담당자가 계속 남아 있을 필요가 없습니다. 이렇게 하면 메모리를 아낄 수 있고, 실수로 두 번 초기화되는 사고도 방지됩니다.

    PART 1 · M3 (2/2)

    3단계 — 설계 결정 풀어보기

    설계 포인트왜 이렇게 했나나중에 어떻게 활용되나
    딱 1번만 실행되고 사라짐 초기화는 앱 시작 때 한 번만 필요. 끝나면 스스로 사라져 메모리를 낭비하지 않음 새로운 초기화 작업(예: 서버 데이터 받아오기)을 이 단계에 그냥 추가하면 됨
    여러 일을 동시에 처리하는 방식(UniTask) 저장 파일 읽는 동안 다른 준비도 동시에 할 수 있어 로딩이 빠름. 앱이 갑자기 종료돼도 작업이 안전하게 정리됨 광고 불러오기, 서버 업데이트 확인 같은 비동기 작업도 같은 방식으로 추가 가능
    저장값 + 1 = 현재 도전 스테이지 "5단계까지 클리어했다"는 저장값으로부터 "다음 도전은 6단계"라고 변환하는 로직을 이 담당자가 한 곳에서만 처리 저장 방식이 바뀌어도 이 변환 로직만 수정하면 됨 — 화면 담당자는 영향받지 않음
    비동기 방식으로 메인 화면 이동 로딩 진행률 표시나 로딩 애니메이션을 나중에 추가하기 쉬움 스플래시 화면, 로딩 진행 바 등을 이 단계에서 추가 가능

    확장 예시: 나중에 서버에서 패치 데이터를 받아오거나, A/B 테스트 설정을 가져오거나, 광고 SDK를 초기화하는 작업들을 모두 이 초기화 단계 안에서 처리할 수 있습니다. 메인 화면이 뜨기 전에 모든 준비가 완료되는 구조입니다.

    PART 1 · M4 (1/2)

    4단계 완료 화면 전환 관리자 & 안전 영역

    핵심 아이디어: 화면을 통째로 바꾸는 대신, 5개의 패널(화면 영역)을 켜고 끄는 방식으로 전환합니다. 이렇게 하면 화면 전환 비용이 거의 0이고, 패널 간에 데이터를 주고받기도 쉽습니다.

    🖼 화면 전환 관리자 (UIManager)

    • 5개 화면(메인·게임·클리어·실패·게임완료)을 관리
    • "메인 화면 보여줘" 명령 한 번으로 나머지는 자동 비활성화
    • 현재 스테이지 번호를 화면에 표시하는 기능
    • 플레이 버튼 / 처음부터 다시 버튼 이벤트 처리
    • 게임 전체에서 딱 1개만 존재하는 관리자

    📐 안전 영역 자동 조정 (SafeAreaHandler)

    • 아이폰처럼 카메라 구멍(노치)이나 하단 홈바가 있는 폰 자동 대응
    • 게임 UI가 그 영역에 가려지지 않도록 화면 영역을 실시간 조정
    • 코드 한 줄 없이 컴포넌트를 붙이는 것만으로 모든 기기에 자동 적용
    • 일반 베젤 폰에서는 영향 없음 (전체 화면 그대로 사용)

    다음 슬라이드에서 5개 화면과 각 전환 조건을 자세히 설명합니다.

    PART 1 · M4 (2/2)

    4단계 — 5개 화면과 전환 조건

    화면 이름어떤 화면인가언제 표시되나
    메인 화면
    MainPanel    		 타이틀 + 현재 스테이지 번호 + 플레이 버튼이 있는 시작 화면 앱 시작 직후 / 클리어 또는 실패 후 다음 도전 전 / "처음부터 다시" 버튼 클릭 후
    게임 화면
    InGamePanel    		 실제 게임이 진행되는 공간 (현재는 테스트용 클리어/실패 버튼) 메인 화면에서 플레이 버튼을 눌렀을 때
    클리어 화면
    ClearPanel    		 스테이지 성공 시 나타나는 화면. M6 완료: 페이드인 + 텍스트 스케일 인 연출 적용 게임 화면에서 클리어 판정이 발생했을 때 (일반 스테이지)
    실패 화면
    FailPanel    		 스테이지 실패 시 나타나는 화면. M6 완료: 페이드인 + 텍스트 흔들기(shake) 연출 적용 게임 화면에서 실패 판정이 발생했을 때
    게임 완료 화면
    GameClearPanel    		 20스테이지를 모두 클리어했을 때의 엔딩 화면 + 처음부터 다시 버튼 20스테이지 클리어 판정 시

    안전 영역(Safe Area) 비유: 아이폰의 카메라 구멍(노치)을 마스킹 테이프로 가려놨다고 생각해보세요. SafeAreaHandler는 그 테이프 바깥에만 UI를 그리도록 자동으로 영역을 잡아줍니다. 개발자가 기기마다 따로 조정할 필요가 없습니다.

    PART 1 · M5 (1/2)

    5단계 완료 게임 진행 컨트롤러

    게임 교통 정리 담당자(GameFlowController) — 클리어·실패 신호를 받아서 다음 화면으로 이동시키는 지휘자

    🏆 클리어 신호 수신 ↓ 세이브 파일 담당자에게 저장 요청 (즉시!) ↓ 마지막 스테이지(20단계)? ├─ 예 → 게임 완료 화면으로 └─ 아니오 → 다음 스테이지로 +1 ↓ 화면의 스테이지 번호 갱신 ↓ 메인 화면으로
    💥 실패 신호 수신 ↓ 저장 없음 (스테이지 번호 그대로 유지) ↓ 화면의 스테이지 번호 확인 (변화 없음) ↓ 메인 화면으로 복귀 (같은 스테이지에서 다시 도전)

    🔑 저장을 먼저 하는 이유

    클리어 직후 바로 저장합니다. 연출 화면이 나오는 도중에 앱이 꺼져도 진행 데이터가 날아가지 않습니다. "저장 먼저, 연출 나중"이 원칙입니다.

    🔑 6단계(M6) 자연스러운 합류

    5단계에서 비워뒀던 "연출 자리"에 6단계에서 애니메이션 담당자(AnimationController)가 끼어 들어왔습니다. 콜백(callback) 방식으로 연결되어 연출이 끝나면 자동으로 다음 화면으로 넘어갑니다. M6 완료

    PART 1 · M5 (2/2)

    5단계 — 구현 완료 내역

    작업무엇을 했나결과
    게임 교통 정리 담당자 신규 생성 클리어·실패 판정 처리, 저장, 다음 화면 이동 로직을 담은 새 파일 작성. 메인 화면 전용으로 딱 1개만 존재 완료 Play Mode 3가지 시나리오 전부 통과
    클리어 버튼 연결 게임 화면의 "클리어" 버튼 클릭 → 게임 교통 정리 담당자의 클리어 처리 메서드 호출 고정 등록 완료 씬 파일에 영구 저장됨
    실패 버튼 연결 게임 화면의 "실패" 버튼 클릭 → 실패 처리 메서드 호출 고정 등록 완료 씬 파일에 영구 저장됨
    "처음부터 다시" 버튼 연결 게임 완료 화면의 "처음부터 다시" 버튼 → 저장 초기화 → 1단계 메인 화면으로 복귀 완료 기존 화면 관리자 재사용

    이 설계의 핵심 재사용 포인트: 나중에 실제 게임(퍼즐, 리듬 등)을 추가할 때 "클리어 조건이 충족되면 → 게임 교통 정리 담당자에게 클리어 신호를 보낸다"만 구현하면 됩니다. 저장·화면 전환·연출은 이미 완성되어 있어서 건드릴 필요가 없습니다.

    ✅ "연출 없는 완전한 MVP" 완성 — 앱 시작부터 20단계 클리어까지 끊김 없이 동작. 6단계(M6)에서 DOTween 연출이 합류하여 MVP가 시각적으로도 완성되었습니다.

    PART 1 · M6 (1/2)

    6단계 완료 클리어·실패 화면 연출

    애니메이션 담당자(AnimationController) — 부드러운 연출을 재생하고 끝나면 신호를 보내는 전담 담당자

    🏆 클리어 연출 — 약 1.4초

    • 밝은 오버레이가 서서히 나타남 (0.4초 페이드인)
    • "CLEAR" 텍스트가 작은 점에서 통통 튀어 나오듯 커짐
    • 1초간 멈춰서 보여준 뒤 다음 스테이지로 자동 이동

    💥 실패 연출 — 약 1.3초

    • 붉은 오버레이가 서서히 나타남 (0.4초 페이드인)
    • "FAIL" 텍스트가 좌우로 흔들리는 효과 (0.5초)
    • 0.8초간 멈춰서 보여준 뒤 같은 스테이지로 자동 복귀

    지휘자 비유: 게임 교통 정리 담당자(지휘자)는 연출이 몇 초짜리인지 알 필요가 없습니다. "연출 끝났으면 알려줘"라고 부탁만 하고, 애니메이션 담당자가 끝났을 때 신호(callback)를 돌려보내는 구조입니다. 이렇게 하면 나중에 연출 시간을 변경해도 흐름 코드를 건드릴 필요가 없습니다.

    PART 1 · M6 (2/2)

    6단계 — 개발자 편의 도구 추가

    🛠 자동 진입점 보정 (SceneBootstrapper)

    • 개발 중에 메인 화면(Main 씬)에서 직접 Play 버튼을 눌러도 자동으로 Boot 화면부터 시작
    • "세이브 데이터 불러오기"가 항상 동작하므로 테스트 편의성 향상
    • 실제 출시 빌드에서는 자동으로 제거되어 영향 없음 (에디터 전용)

    🔧 위임(delegation) 방식의 깔끔한 구조

    • 게임 교통 정리 담당자 → 화면 전환 관리자 → 애니메이션 담당자 순으로 연결
    • 각 담당자는 자기 역할에만 집중 (단방향 위임)
    • 나중에 연출 종류를 늘리거나 바꿔도 흐름 코드는 건드릴 필요 없음

    왜 5단계에서 흐름을 먼저 검증하고 6단계에서 연출을 얹었나? — 흐름 버그와 연출 버그가 섞이면 디버깅이 어렵습니다. "스테이지가 제대로 넘어가는가"를 5단계에서 완전히 확인한 뒤, 6단계에서 연출만 얹으니 연출 관련 문제만 따로 추적할 수 있었습니다. 결과적으로 6단계는 빠르게 완료되었습니다.

    PART 1 · 정리

    지금 실제로 동작하는 전체 흐름 (6단계 완료 기준)

    [앱 실행] ↓ 초기화 전문가(BootLoader) 실행 ↓ 세이브 파일 담당자 → "저장된 최고 단계" 읽기 ↓ 진행 상황 기록원 → 다음 도전 스테이지 설정 ↓ 메인 화면으로 이동 (초기화 담당자 사라짐) ↓ 화면 전환 관리자(UIManager) — 안전 영역 자동 조정 완료 ↓ 스테이지 번호 화면에 표시 ↓ 메인 화면 활성화 ↓ [플레이 버튼 클릭] → 게임 화면으로 [클리어 버튼 클릭] → 게임 교통 정리 담당자(GameFlowController) ├── 즉시 저장 └── 화면 전환 관리자 → 애니메이션 담당자 ├── 클리어 연출 재생 (~1.4초) └── 콜백 → 20단계? → 게임 완료 화면 → 아니면 다음 단계 메인 화면 [실패 버튼 클릭] → 화면 전환 관리자 → 애니메이션 담당자 ├── 실패 연출 재생 (~1.3초) └── 콜백 → 저장 없이 같은 단계 메인 화면 복귀 [처음부터 다시] → 저장 초기화 → 1단계 메인 화면 복귀

    ✅ 6단계까지 완료 — 클리어/실패 버튼 클릭부터 부드러운 연출 재생, 자동 화면 전환까지 끊김 없이 동작하는 시각적으로도 완성된 MVP입니다. 다음 7단계에서 코드 내부 정리와 안정성 보강을 진행합니다.

    초기화 전문가(BootLoader) — Boot 화면 전용 ├── 세이브 파일 담당자(SaveManager)에게 저장값 요청 ✅ └── 진행 상황 기록원(StageManager)에게 스테이지 전달 ✅ 게임 교통 정리 담당자(GameFlowController) — 메인 화면 전용 ├── 진행 상황 기록원(StageManager) ✅ — 다음 단계로 넘기기, 마지막 단계 확인 ├── 화면 전환 관리자(UIManager) ✅ — 연출 재생 위임 + 화면 전환 └── 세이브 파일 담당자(SaveManager) ✅ — 클리어 직후 저장 요청 화면 전환 관리자(UIManager) — 메인 화면 전용 ├── 진행 상황 기록원(StageManager) ✅ — 스테이지 번호 읽기 ├── 세이브 파일 담당자(SaveManager) ✅ — 처음부터 다시 시 초기화 └── 애니메이션 담당자(AnimationController) ✅ — 클리어/실패 연출 위임 (M6) 애니메이션 담당자(AnimationController) ✅ M6 완료 — DOTween만 사용 (다른 담당자 의존 없음) 안전 영역 조정기(SafeAreaHandler) ✅ 독립적으로 동작 진행 상황 기록원(StageManager) ✅ 독립적으로 동작 세이브 파일 담당자(SaveManager) ✅ 독립적으로 동작 자동 진입점 보정기(SceneBootstrapper) ✅ 에디터 전용 유틸 (의존 없음)

    (라인 잇기 등)    		 게임 화면에 퍼즐 판 추가. 클리어 조건 충족 시 게임 교통 정리 담당자에게 클리어 신호 전송 저장·불러오기, 다음 스테이지 이동, 클리어·실패 화면 전환은 모두 자동 처리

    아케이드
    (피하기·점프)    		 게임 화면에 캐릭터·장애물 시스템 추가. 사망 시 실패 신호 전송 스테이지 데이터 파일에 난이도(속도·패턴) 필드만 추가하면 20단계 난이도 자동 구성
    퀴즈 게임 스테이지 데이터 파일에 질문·정답 필드 추가. 정답이면 클리어 신호, 오답이면 실패 신호 20문제 한 세트 자동 구성. 오답 시 같은 문제 재시도, 20문제 전부 정답 시 완료 화면

    public class SaveManager : MonoBehaviour
{
    private const string SAVE_KEY =
        "MaxClearedStage";

    public void Save(int maxClearedStage)
    {
        PlayerPrefs.SetInt(SAVE_KEY, maxClearedStage);
        PlayerPrefs.Save();
    }

    public int Load() =>
        PlayerPrefs.GetInt(SAVE_KEY, 0);

    public void Reset()
    {
        PlayerPrefs.DeleteKey(SAVE_KEY);
        PlayerPrefs.Save();
    }
}

    설계 포인트선택 이유대안
    싱글턴 아님Boot 씬과 생사를 같이하므로 DontDestroyOnLoad 불필요. 중복 실행 위험 없음싱글턴으로 만들면 씬 전환 후 남아서 메모리 낭비
    Start()에서 시작 M6 변경모든 매니저의 Awake() 완료 후 실행 보장 → SaveManager.Instance null 참조 방지Awake에서 직접 호출 시 매니저 초기화 순서 의존
    UniTask + CancellationToken씬 전환 도중 앱 종료 시 안전하게 작업 취소 가능코루틴: 취소 신호 전파 어려움
    InitializeAsync 분리진입점을 단순하게 유지, 테스트 시 직접 호출 가능Start 직접 구현: 테스트 어렵고 가독성 저하
    OnDestroy에서 _cts.Cancel()씬 전환 중 오브젝트 파괴 시 진행 중인 UniTask가 예외 없이 종료됨미처리 시 OperationCanceledException 발생 가능
     
    private CancellationTokenSource _cts;

// M6: Awake → Start로 이동
// 모든 매니저 Awake 완료 후 호출 보장
void Start()
{
    _cts = new CancellationTokenSource();
    InitializeAsync(_cts.Token).Forget();
}

private async UniTask InitializeAsync(
    CancellationToken ct)
{
    int maxCleared = SaveManager.Instance.Load();
    StageManager.Instance
        .SetCurrentStage(maxCleared + 1);
    await SceneManager
        .LoadSceneAsync("Main")
        .WithCancellation(ct);
}

void OnDestroy()
{
    _cts?.Cancel();
    _cts?.Dispose();
}

    PART 3 · GameFlowController

    GameFlowController — 오케스트레이터

    메서드흐름 M6 통합
    OnStageClear() ① SaveManager.Save(현재) 즉시 저장 → ② UIManager.PlayClearAndThen(콜백) → ③ 콜백: IsLastStage()? → ③-a ShowPanel(GameClear) / ③-b AdvanceToNextStage → RefreshStageDisplay → ShowPanel(Main)
    OnStageFail() ① UIManager.PlayFailAndThen(콜백) → ② 콜백: RefreshStageDisplay → ShowPanel(Main) (스테이지 번호 변경 없음)

    M6 통합 완료: 콜백(Action) 기반 위임 방식으로 구현됨. GFC는 애니메이션 길이·종류를 몰라도 되고, AnimationController는 호출자의 후속 처리 로직을 몰라도 된다. UIManager가 중간 위임 계층 역할을 하므로 GFC와 AnimationController 사이 직접 의존이 발생하지 않는다.

    public void OnStageClear()
{
    // 저장 먼저 — 연출 중 앱 종료 대비
    SaveManager.Instance.Save(
        StageManager.Instance
            .CurrentStageNumber);

    // 클리어 연출 위임 + 완료 콜백
    UIManager.Instance.PlayClearAndThen(() =>
    {
        if (StageManager.Instance.IsLastStage())
        {
            UIManager.Instance
                .ShowPanel(PanelType.GameClear);
        }
        else
        {
            StageManager.Instance
                .AdvanceToNextStage();
            UIManager.Instance
                .RefreshStageDisplay();
            UIManager.Instance
                .ShowPanel(PanelType.Main);
        }
    });
}

public void OnStageFail()
{
    UIManager.Instance.PlayFailAndThen(() =>
    {
        UIManager.Instance.RefreshStageDisplay();
        UIManager.Instance
            .ShowPanel(PanelType.Main);
    });
}
    PART 3 · AnimationController

    AnimationController — DOTween Sequence 위임자 M6 신규

    메서드Sequence 구성
    PlayClear(Action onComplete) _clearOverlay.DOFade(0.85f, 0.4f) + _clearTextTransform.DOScale(1f, 0.4f).From(0).SetEase(OutBack) → AppendInterval(1.0f) → OnComplete(콜백)
    PlayFail(Action onComplete) _failOverlay.DOFade(0.85f, 0.4f) + _failTextTransform.DOShakePosition(0.5f, 20f) → AppendInterval(0.8f) → OnComplete(콜백)
    SerializeField역할
    _clearOverlay / _failOverlayCanvasGroup — 페이드 인/아웃 대상
    _clearTextTransform / _failTextTransformTransform — 스케일/흔들기 대상
    _activeSequence (private)이전 Sequence가 살아있으면 Kill()로 강제 종료 (중복 재생 방지)
     
    public void PlayClear(Action onComplete)
{
    _activeSequence?.Kill();

    _activeSequence = DOTween.Sequence()
        .Join(_clearOverlay
            .DOFade(0.85f, 0.4f)
            .From(0f))
        .Join(_clearTextTransform
            .DOScale(1f, 0.4f)
            .From(Vector3.zero)
            .SetEase(Ease.OutBack))
        .AppendInterval(1.0f)
        .OnComplete(() => onComplete?.Invoke());
}

public void PlayFail(Action onComplete)
{
    _activeSequence?.Kill();

    _activeSequence = DOTween.Sequence()
        .Join(_failOverlay
            .DOFade(0.85f, 0.4f)
            .From(0f))
        .Join(_failTextTransform
            .DOShakePosition(0.5f, 20f))
        .AppendInterval(0.8f)
        .OnComplete(() => onComplete?.Invoke());
}

    설계 의도: AnimationController는 "재생 + 완료 시 콜백 호출"만 담당한다. 어떤 화면으로 넘어갈지, 다음 스테이지가 무엇인지 모른다 — 호출자(UIManager → GameFlowController)가 콜백 안에서 결정한다. 이렇게 책임을 분리하면 연출 종류가 늘어도 GFC는 영향을 받지 않는다. _activeSequence?.Kill() 패턴으로 빠른 연속 클릭 시 잔여 Tween이 누적되는 문제를 방지한다.
