    6 / 32 · M1 — 프로젝트 기반

    M1 · 프로젝트의 뼈대 만들기

    폴더 구조

    스크립트, 화면 설정, 씬, 데이터 파일을 종류별로 정리할 서랍을 먼저 만들었습니다. 나중에 부품이 100개 넘어가도 어디서 찾을지 헷갈리지 않게요.

    화면 해상도 기준

    가로 1080 × 세로 1920 픽셀을 기준 크기로 정했습니다. 다른 크기 화면(아이폰·갤럭시·태블릿)에서도 자동으로 비율 맞춰 보이도록 설정했습니다.

    왜 맨 처음에 이걸 했나요?
    화면 기준을 늦게 바꾸면 이미 배치한 모든 버튼·텍스트를 다시 옮겨야 합니다. 집을 다 지은 뒤에 "1층을 30cm 올릴까?"를 결정하는 것과 같죠. 그래서 가장 먼저 정합니다.
    7 / 32 · M2 — 데이터 & 저장

    M2 · 스테이지 정보와 진행 기록

    스테이지 데이터 파일

    각 스테이지가 몇 번인지, 어떤 이름인지, 어떤 연출이 재생될지를 적은 설정 카드 20장을 만들었습니다.

    진행 상황 기록원

    "지금 몇 번째 스테이지인지"를 항상 기억하는 담당자입니다. 메인 화면에 "STAGE 5"라고 띄울 때 이 담당자에게 물어봅니다.

    세이브 파일 담당자

    "몇 번까지 깼는지"를 앱이 꺼져도 기억하도록 메모해 두는 담당자입니다. 앱을 다시 켜면 이 메모를 읽어서 이어서 시작하게 합니다.

    왜 데이터 담당자부터 만들었나요?
    화면을 만들기 전에 "보여줄 정보"가 먼저 있어야 합니다. 신문을 만들 때 글을 먼저 쓰고 인쇄소에 넘기는 것과 같죠. 데이터가 준비되면 화면 만들기가 훨씬 쉬워집니다.
    8 / 32 · M2 — 데이터 & 저장

    각 담당자가 하는 일

    담당자역할실제로 언제 쓰이나
    스테이지 데이터 파일각 스테이지의 고정 정보 보관 (번호, 이름, 연출 종류)화면에 스테이지 정보를 표시할 때
    진행 상황 기록원현재 도전 중인 스테이지 번호 기억, 다음 스테이지로 넘기기, 1~20 범위 강제"STAGE 5" 글씨 표시, 클리어 후 +1
    세이브 파일 담당자"최고 클리어 스테이지" 한 가지 숫자만 저장·불러오기·초기화앱 시작·클리어 직후·다시 시작 누를 때
    왜 세이브를 캡슐화했나요?
    저장 방법은 나중에 바뀔 수 있습니다. 처음엔 메모장(PlayerPrefs)이지만, 나중에 클라우드 동기화를 추가할 수도 있습니다. 그때 게임의 다른 부분을 안 건드려도 되도록 "세이브 담당자"한 명에게만 저장 방식을 알려줍니다.
    9 / 32 · M3 — Boot 씬 진입

    M3 · 앱 시작 시 가장 먼저 일어나는 일

    앱 실행 → 초기화 전문가 호출 → 세이브 파일 확인 → 스테이지 번호 설정 → 메인 화면 띄우기

    초기화 전문가 (BootLoader)

    앱이 켜지자마자 0.5초 안에 다음 일을 끝내는 담당자입니다.

    1. 세이브 파일 담당자에게 "마지막 클리어 몇 번?"을 물어봄
    2. 진행 상황 기록원에게 "그 다음 번호로 설정해줘"라고 지시
    3. 메인 화면이 있는 씬으로 화면을 전환
    4. 자기 자신은 사라짐 (역할 끝)
    "사라지는 담당자"라는 게 무슨 뜻인가요?
    대부분의 담당자는 게임이 끝날 때까지 살아 있지만, 초기화 전문가는 일이 끝나면 자기 자신을 정리합니다. 호텔 체크인 직원처럼 "처음에만 필요한 사람"이라 메모리를 차지하지 않게 설계했습니다.
    10 / 32 · M3 — Boot 씬 진입

    왜 초기화 전문가를 따로 두었나

    설계 포인트왜 이렇게 했나나중에 어떻게 확장될까
    Boot 씬을 별도로 만들기"앱이 시작되는 진입점"을 명확히 하나로 고정광고 SDK, 분석 도구 등 시작 시점 작업을 여기에 추가
    한 번 쓰고 사라지게메인 화면으로 가면 더 이상 필요 없음 — 메모리 절약로딩 진행률 표시, 스플래시 화면 등을 자연스럽게 끼워 넣기
    비동기로 설계 (UniTask)저장 읽기·씬 로드를 동시에 처리할 수 있고 끊김 없음네트워크 인증, 클라우드 세이브 등 시간 걸리는 작업 추가 시 안전
    비유 — 비행기 이륙 전 점검.
    승무원이 안전벨트·비상구·산소마스크를 안내하는 시간이 "Boot 씬"입니다. 이륙 후엔 안내방송이 사라지죠. 게임도 똑같습니다. 시작에만 필요한 일을 따로 두면 본 게임이 깔끔해집니다.
    11 / 32 · M4 — 화면 전환

    M4 · 5개의 화면을 깔끔하게 갈아끼우기

    화면 전환 관리자 (UIManager)

    5개의 화면(메인·인게임·클리어·실패·완료)을 한 명의 담당자가 관리합니다. "지금 메인 화면 보여줘"라고 말하면 다른 화면들은 자동으로 숨깁니다.

    5개 화면 = 5개 패널 (한 씬 안에 모두 배치)

    안전 영역 처리자 (SafeAreaHandler)

    아이폰의 노치나 홈바에 버튼이 가려지지 않도록 자동으로 여백을 잡아주는 담당자입니다. 기기마다 다른 안전 영역을 알아서 맞춰줍니다.

    왜 씬 전환 없이 패널만 갈아끼우나요?
    새 씬을 로드할 때마다 1~2초씩 멈추거나 검은 화면이 보입니다. 한 씬 안에서 패널만 켜고 끄면 즉시 전환이 가능합니다. 모바일 캐주얼 게임에선 이 즉시성이 매우 중요합니다.
    12 / 32 · M4 — 화면 전환

    각 화면이 언제 나타나나

    화면어떤 상황에서전환 조건
    메인 화면앱 시작 직후, 클리어/실패 연출 끝난 후플레이 버튼을 누르면 인게임으로
    인게임 화면실제 게임 진행 중클리어 또는 실패 버튼 클릭
    클리어 화면스테이지 클리어 직후1.4초 연출 후 다음 메인으로 자동
    실패 화면스테이지 실패 직후1.3초 연출 후 동일 메인으로 자동
    게임 완료 화면20스테이지 클리어 시다시 시작 버튼으로 메인 복귀
    안전 영역(Safe Area) 비유 — 액자에 그림을 끼울 때 종이 끝까지 그림이 닿으면 액자에 가려집니다. 그래서 가장자리에 여백을 둡니다. 휴대폰의 노치·홈바도 똑같이 화면 일부를 가리므로, 미리 그 영역만큼 비워두는 것입니다.
    13 / 32 · M5 — 판정 로직

    M5 · 클리어/실패 후 어디로 갈지 정하는 담당자

    게임 교통 정리 담당자 (GameFlowController)

    인게임 화면에서 클리어/실패 판정이 발생하면 교차로 신호등처럼 다음 행선지를 결정합니다. 자기는 직접 화면을 바꾸지 않고, 다른 담당자에게 "이렇게 해줘"라고 지시만 합니다.

    클리어 버튼→ 즉시 저장→ 클리어 연출→ 20번?→ 예 / 게임 완료| 아니오 / 다음 메인
    실패 버튼→ 실패 연출→ 동일 스테이지 메인
    왜 저장이 연출 전에 이루어지나요?
    만약 연출 도중에 앱이 갑자기 종료되더라도 클리어 기록은 남아 있어야 하기 때문입니다. "성공은 무조건 즉시 기록, 그 다음에 보여주기"가 안전한 순서입니다.
    14 / 32 · M5 — 판정 로직

    M5에서 완성된 기능

    작업무엇을 했나결과
    판정 로직 본체클리어/실패 후 저장·다음 스테이지·화면 전환을 한 흐름으로 묶음1→2→3 자동 진행 확인
    버튼 연결인게임의 초록·빨강 버튼을 판정 메서드에 직접 연결버튼 누르면 즉시 작동
    20스테이지 분기마지막 스테이지에서는 다음 스테이지가 아닌 게임 완료 화면으로20번 클리어 후 완료 화면 표시
    다시 시작 버튼완료 화면의 버튼이 저장 초기화 + 1스테이지부터 다시전체 리셋 PASS
    프로토타입 버튼은 어떻게 제거하나요?
    실제 게임 규칙(예: 퍼즐 해결)이 들어오면, 그 규칙이 직접 클리어/실패 메서드를 호출합니다. 버튼만 떼어내면 되고 판정 로직은 그대로 재사용됩니다. 이게 미리 분리해 둔 이유입니다.
    15 / 32 · M6 — 클리어/실패 연출

    M6 · 부드럽게 등장하고 흔들리는 연출

    클리어 연출 (성공)

    오버레이가 천천히 어두워지면서(0.4초) "CLEAR!" 글씨가 탱탱하게 튀어나옵니다. 1초 정도 보여준 뒤 다음 화면으로.

    총 1.4초 — 성취감을 주는 시간

    실패 연출 (실패)

    오버레이가 어두워지면서 "FAILED" 글씨가 짧게 흔들립니다(0.5초). 0.8초 후 같은 스테이지로 복귀.

    총 1.3초 — 짧고 명확한 피드백

    연출 전담 담당자(AnimationController)를 따로 둔 이유.
    교통 정리 담당자(GameFlowController)는 "어디로 갈지"만 결정하고, "어떻게 보여줄지"는 연출 담당자에게 맡깁니다. 그래서 나중에 연출을 바꾸고 싶을 때(예: 별점 표시 추가) 다른 담당자를 건드릴 필요가 없습니다.
    16 / 32 · M6 — 클리어/실패 연출

    지휘자처럼 일을 시키는 흐름

    교통 정리 담당자 → 화면 전환 관리자에 위임 → 클리어 화면 표시 → 연출 담당자에 위임
    연출 1.4초 재생 → 연출 끝나면 "다 끝났어요" 알림 → 다음 화면 결정 (분기)

    왜 "끝나면 알려줘" 방식을 썼나

    지휘자가 트럼펫 연주자에게 "8마디 연주해" 라고 말하면, 연주자가 끝났을 때 다음 신호를 줍니다. 지휘자는 8마디가 정확히 몇 초인지 몰라도 됩니다. 이걸 콜백 체인이라고 부릅니다.

    실용 효과 — 연출 시간을 1.4초에서 2초로 늘려도 교통 정리 담당자의 코드는 한 줄도 안 바뀝니다. 연출 담당자만 수정하면 됩니다.
    17 / 32 · M7 — 안정성·성능

    M7 · 100스테이지로 늘려도 견디는 코드 만들기

    왜 했나

    현재 20스테이지에서는 잘 돌아가지만, 만약 게임이 잘되어 100스테이지·500스테이지로 늘어나면 지금 코드는 조금씩 느려집니다. 그렇게 되기 전에 미리 정리하는 작업입니다.

    사용자에게 보이는 차이

    거의 없습니다. 외관이나 기능은 그대로입니다. 하지만 개발자에겐 "코드를 고치다가 실수해서 망가지는 일"이 줄어듭니다.

    단계주제한 줄 요약
    M7-A안전망 추가잘못 설정해도 에러 메시지가 명확히 뜨도록
    M7-B코드 청소안 쓰는 부품 제거, 텍스트는 한 곳에 모음
    M7-C화면 전환 속도패널이 100개로 늘어나도 즉시 전환
    M7-D저장·로딩 안전성매 프레임 호출돼도 빠름, 저장 중에도 끊김 없음
    비유 — 정기 점검 받은 자동차.
    주행감은 똑같지만 부품을 미리 점검·교체하면 멀리 가도 멈추지 않습니다. M7도 같은 성격의 작업입니다.

    SaveManager — PlayerPrefs 캡슐화

    메서드내부 구현
    Save(int)PlayerPrefs.SetInt(SAVE_KEY, value) + Save() (동기)
    SaveAsync(int): UniTask (M7-D)UniTask.RunOnThreadPool로 백그라운드 저장 — 모바일 I/O 프리징 회피
    Load(): intPlayerPrefs.GetInt(SAVE_KEY, 0)
    Reset()PlayerPrefs.DeleteKey(SAVE_KEY) + Save()
     
    // SaveManager.cs — DontDestroyOnLoad 싱글턴, PlayerPrefs 직접 노출 금지
public class SaveManager : MonoBehaviour {
    private const string SAVE_KEY = "MaxClearedStage";
    public static SaveManager Instance { get; private set; }

    public void Save(int maxClearedStage) {
        PlayerPrefs.SetInt(SAVE_KEY, maxClearedStage);
        PlayerPrefs.Save();
    }

    public async UniTask SaveAsync(int maxClearedStage) {
        await UniTask.RunOnThreadPool(() => Save(maxClearedStage));
    }

    public int Load() => PlayerPrefs.GetInt(SAVE_KEY, 0);
}
    왜 PlayerPrefs를 직접 쓰지 않고 감쌌나 — 향후 SQLite, JSON 파일, 클라우드 동기화로 교체 시 SaveManager 내부만 수정하면 호출자는 무영향. M7-D의 SaveAsync 추가도 같은 캡슐화 덕분에 안전했음.

    설계 포인트선택 이유대안 / 메모

    UniTask + CancellationToken씬 전환 중 안전한 취소코루틴 대비 async/await 가독성 ↑
    Awake → Start 변경 (M6)모든 매니저 Awake 완료 보장StageManager.Instance NRE 방지

    // BootLoader.cs — Boot 씬 전용, M7-D WaitUntil 동기화
public class BootLoader : MonoBehaviour {
    private const string MAIN_SCENE_NAME = "Main";
    private CancellationTokenSource _cts = new();

    private void Start() => InitializeAsync(_cts.Token).Forget();

    private async UniTask InitializeAsync(CancellationToken ct) {
        int maxCleared = SaveManager.Instance.Load();
        StageManager.Instance.SetCurrentStage(maxCleared + 1);
        await SceneManager.LoadSceneAsync(MAIN_SCENE_NAME).WithCancellation(ct);
        await UniTask.WaitUntil(() => UIManager.Instance != null, cancellationToken: ct);
    }

    private void OnDestroy() { _cts?.Cancel(); _cts?.Dispose(); }
}

public enum PanelType { Main, InGame, Clear, Fail, GameClear }

private Dictionary<PanelType, GameObject> _panelMap;
private PanelType? _currentPanelType = null;
private const string STAGE_PREFIX = "STAGE ";  // M7-B

public void ShowPanel(PanelType type) {
    if (_currentPanelType == type) return;  // M7-C early return
    foreach (var kv in _panelMap) kv.Value.SetActive(false);
    _panelMap[type].SetActive(true);
    _currentPanelType = type;
}

    메서드호출 흐름 (M6 콜백 체인)
    OnStageClear()① SaveManager.Save(currentStage) ② UIManager.PlayClearAndThen(콜백) ③ 콜백: IsLastStage ? ShowPanel(GameClear) : AdvanceToNextStage + RefreshStageDisplay + ShowPanel(Main)
    OnStageFail()① UIManager.PlayFailAndThen(콜백) ② 콜백: RefreshStageDisplay + ShowPanel(Main) (스테이지 번호 미변경, 저장 미발생)
     
    // GameFlowController.cs — Main 씬 루트 GameObject, Canvas 외부
public void OnStageClear() {
    int stage = StageManager.Instance.CurrentStageNumber;
    SaveManager.Instance.Save(stage);  // 연출 전에 즉시 저장

    UIManager.Instance.PlayClearAndThen(() => {
        if (StageManager.Instance.IsLastStage()) {
            UIManager.Instance.ShowPanel(PanelType.GameClear);
        } else {
            StageManager.Instance.AdvanceToNextStage();
            UIManager.Instance.RefreshStageDisplay();
            UIManager.Instance.ShowPanel(PanelType.Main);
        }
    });
}

    _clearTextTransform / _failTextTransformTransform, 인스펙터 할당
    _activeSequenceSequence, 재시작 시 Kill() 호출로 중복 재생 방지
     
    // AnimationController.cs — UIManager 위임 호출만 받음
public void PlayClear(Action onDone) {
    _activeSequence?.Kill();
    _clearOverlay.alpha = 0;
    _clearTextTransform.localScale = Vector3.zero;

    _activeSequence = DOTween.Sequence()
        .Append(_clearOverlay.DOFade(0.85f, 0.4f))
        .Join(_clearTextTransform.DOScale(1f, 0.4f).SetEase(Ease.OutBack))
        .AppendInterval(1.0f)
        .OnComplete(() => onDone?.Invoke());
}

    이제 게임을 만들 차례입니다

    신입 사원이 기억할 것

    • 5개 화면이 있고 한 명의 관리자가 갈아끼움

    • 클리어/실패 후의 행선지는 교통 정리 담당자가 결정

    • 진행 상황은 자동 저장

    • 단방향 의존 — 거꾸로 부르지 않기

    개발자가 다음에 할 일

    • 인게임 콘텐츠 시스템 (퍼즐 / 러너 등)

    • 클리어/실패 판정 규칙을 OnStageClear/Fail에 연결

    • 프로토타입 버튼 제거

    • MVP 이후 기능: 사운드 · 광고 · 리더보드 등

    이 키트의 가치 — 동일한 인프라(앱 시작 · 화면 전환 · 저장 · 연출)를 매번 다시 만들지 않는 것. 게임 한 편 출시할 때마다 절약되는 1~2주가 누적되면 그것이 곧 경쟁력입니다. 다음에 만들어볼 게임 아이디어를 가져오세요.
